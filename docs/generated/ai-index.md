## Rule

Use Host SDK for UI/multi-UI work; use backend RPC/OpenAPI for backend/API contracts.

## Contract Surfaces

| Surface | Use For | Avoid For |
|---|---|---|
| `backend-rpc-describe` | RPC params, responses, permissions, generated OpenAPI, generated Host SDK contract. | UI layout and template-specific convenience APIs. |
| `host-sdk` | Multi-UI/template calls, typed frontend helpers, domain-level UI behavior. | Changing backend API contracts directly. |
| `scalar-openapi` | Scalar docs, OpenAPI clients, future OpenAPI-backed MCP execution. | Repo architecture, frontend call ownership, or internal plans. |
| `local-docs-mcp` | Read-only doc search and exact contract lookup. | Unreviewed write/API execution. |

## Workflows

### Build or change UI/template code

Read: `separate-ui-host-sdk-guide`, `host-sdk`, `host-sdk-index`, `host-contract`, `frontend-backend-dependencies`

The Host SDK is the stable UI-facing layer; raw backend RPC is lower level.

### Install or deploy Fleet Manager

Read: `deployment-guide`, `deploy-reference`, `ai-and-mcp-reference`

The deploy docs define deploy.sh commands, flags, environments, modes, and routing behavior.

### Change backend RPC contract

Read: `api-catalog`, `rpc-inventory`, `openapi`, `auth-matrix`

Backend contract changes must preserve params, responses, and permission boundaries.

### Find existing callers

Read: `frontend-backend-dependencies`, `rpc-inventory`

Caller lookup prevents breaking existing frontend flows.

### Prepare MCP context

Read: `ai-and-mcp-reference`, `ai-index`, `api-catalog`, `openapi`, `rpc-inventory`, `deployment-guide`, `host-contract`

Use read-only resources for context, then authenticated live tools for governed Fleet Manager operations.
